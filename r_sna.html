Let’s time how long it takes to model. For brevity, I’m going to suppress messages that the ergm() command usually outputs (I am also pumping up the MCMC.samplesize to demonstrate how helpful using more cores can be).

system.time(ergm_model <- suppressMessages(
  ergm(molecule ~ edges + kstar(2) + triangle + nodematch("atomic_type"),
       control = control.ergm(MCMC.samplesize = 25000)))
  )
## Warning: `set_attrs()` is deprecated as of rlang 0.3.0
## This warning is displayed once per session.
##    user  system elapsed 
##   45.89    0.06   46.03

Now with two cores.

system.time(ergm_model <- suppressMessages(
  ergm(molecule ~ edges + kstar(2) + triangle + nodematch("atomic_type"),
       control = control.ergm(MCMC.samplesize = 25000, parallel = 2)))
  )
##    user  system elapsed 
##    2.16    0.05   29.95

Finally, with four cores.

system.time(ergm_model <- suppressMessages(
  ergm(molecule ~ edges + kstar(2) + triangle + nodematch("atomic_type"),
       control = control.ergm(MCMC.samplesize = 25000, parallel = 4)))
  )
##    user  system elapsed 
##    2.27    0.03   20.09

As you can see, there are diminishing returns to adding in more cores. That said, using four cores saves a good amount of absolute time, and these gains multiply if your project requires a series of ERGM models.

However, too many cores often consumes more time due to overhead. I have found that it is best to use the number of physical cores your CPU has minus one. For example, all of my ERGM and RSiena code use three or four cores for modeling (depending on if I am multitasking on my desktop), as I currently operate on a heavily-overclocked (5.2 GHz) Intel Core i7-7700K, which is a four-core, eight-thread processor.

Consequently, improving social network analysis performance (and R performance more generally) requires faster CPUs, even if that means fewer cores (sorry AMD fans; at the time of writing, Intel is still king here).

SIENA

RSiena is a very different beast that I cannot hope to explain succinctly. I highly recommend consulting the in-depth manual frequently.

library(RSiena)

siena_results = siena07(
    siena_algorithm,            # Algorithm object created via sienaAlgorithmCreate().
    data       = siena_data,    # Data object created via sienaDependent() and sienaDataCreate().
    effects    = siena_effects, # Effects object created via getEffects() and includeEffects().
    batch      = T, # Disable the graphical user interface.
    returnDeps = T, # Return simulated dependent variables (networks, behavior).
    useCluster = T, # Enable use of cluster.
    nbrNodes   = 4  # Number of processes to use; should match the number of physical cores you have.
)

# Write the results out to an HTML file.
siena.table(siena_results, type = "html", sig = T, tstatPrint = T, d = 3, file = "your_filename_here.htm")

Though RSiena does not return simulated dependent variables by default, you need to set that parameter to true if you want to determine goodness of fit or include diagnostic plots in your research paper. Be advised that this consumes a significant amount of RAM, especially if you increase Phase 3 iterations via the n3 parameter of the sienaAlgorithmCreate() command. Note that you are supposed to do this to achieve stable estimation of standard errors; again, be sure to refer to the documentation.
